# AUBB V1 开发总计划

## 1. 计划定位

本计划基于 [AUBB 软件需求规格说明书 v4.2](../../docs/02-process-docs/software-requirements-specification.md) 编制，用于指导 AUBB V1 从需求冻结、架构设计、数据设计、接口设计、分阶段开发、测试验收到上线准备的完整流程。

本计划面向当前 `AUBB-Server` 后端仓库，因此重点覆盖：

- 后端领域建模、接口与数据持久化
- 提交、评测、成绩、通知、审计等核心业务链路
- CI/CD、可观测性、安全、性能、可靠性等工程能力
- 与前端、评测沙箱、文件存储、统一认证、消息通道等外部系统的协同边界

本计划不替代以下文档：

- 详细 API 文档
- 数据库物理设计文档
- 详细设计说明书
- 前端实施计划
- 运维 Runbook

## 2. 规划原则

1. 以 SRS 的 `MUST / SHOULD / COULD` 作为交付分层依据，先完成 V1 基线，再进入增强项。
2. 以“主链路贯通”为第一优先级，先打通“平台初始化 -> 课程 -> 任务 -> 提交 -> 评测 -> 批改 -> 成绩 -> 查询”。
3. 每个功能阶段必须同时交付代码、测试、文档、接口说明、监控点和审计点，禁止后补。
4. 所有持久化变更必须通过 Flyway 进入仓库，并同步更新 `docs/generated/db-schema.md`。
5. 采用 TDD 与增量集成方式推进，避免先堆大量模块再统一联调。
6. 任何架构级调整都要更新 `ARCHITECTURE.md`、相关 ADR/计划文档和质量评分说明。

## 3. 范围与交付层次

### 3.1 V1 基线

V1 基线目标是实现 SRS 中全部 `MUST` 级功能和非功能要求，形成可交付、可验证、可上线的首版系统。

核心能力包括：

- 平台配置、组织、用户、角色、账号状态管理
- 登录、鉴权、授权、受保护资源访问控制
- 课程、成员、资源、任务、评分规则管理
- 提交受理、在线 IDE 工作区、试运行与正式提交分离
- 自动评测编排、隔离执行、结果结构化输出与持久化
- 人工批改、成绩生成、发布、撤回、结果查看
- 站内通知、课程级概览、审计日志
- MUST 级性能、可靠性、安全、兼容性和可观测性要求

### 3.2 V1.1 优先增强

依据 SRS 第 8.4 节，V1.1 优先增强范围包括：

- 密码管理、统一认证扩展
- 课程结构编排、课程归档历史访问
- Rubric 配置、工作区草稿恢复
- 评测规则配置、重新评测
- 成绩导出、成绩复核、个人学习概览
- 系统公告、多渠道通知扩展
- 平台级概览、异常事件留痕

### 3.3 后续版本候选

- 跨课程资源 / 模板复用
- 任务模板复用
- 多运行环境模板
- 高级反作弊、深度三方集成、AI 能力

## 4. 总体开发流程

### 4.1 标准流程

1. 需求冻结：从 SRS 选定阶段范围，关闭该阶段的未决事项和范围歧义。
2. 设计先行：输出模块边界、状态流转、数据实体映射、接口契约和异常策略。
3. 建库与迁移：先设计逻辑实体，再落 Flyway 迁移和持久化实现。
4. 测试先行：先写领域测试、接口测试、集成测试和关键安全/边界用例。
5. 增量实现：按模块切片完成 API、应用服务、领域模型、基础设施适配器。
6. 联调集成：打通与前端、评测沙箱、文件存储、通知通道、认证源的接口。
7. 质量加固：补性能压测、故障隔离、监控指标、告警、审计和安全加固。
8. 验收上线：完成 UAT、回归、上线检查、回滚方案和运维交接。

### 4.2 每个功能切片的统一交付动作

每一个功能切片都按以下顺序推进：

1. 建立执行计划，标注对应 SRS 编号和验收目标。
2. 补充或更新产品规格、设计说明、ADR、状态图或时序图。
3. 明确 API 契约、实体边界、权限要求和异常场景。
4. 设计并提交 Flyway 迁移、实体映射与数据字典更新。
5. 先写测试：单元测试、接口测试、权限测试、边界测试、必要的集成测试。
6. 实现 `api -> application -> domain -> infrastructure` 切片代码。
7. 接入日志、指标、审计、链路标识和错误上下文。
8. 更新 OpenAPI、仓库文档和操作说明。
9. 执行 `.\mvnw.cmd verify` 与阶段专项验证。
10. 合并前复核风险、测试证据和未决问题。

## 5. 建议模块切片

建议以后端域为中心组织模块，而不是按控制器或表名堆叠：

- `platform-config`：平台配置、品牌、资源策略、配置版本
- `identity-access`：登录、会话、账号状态、角色授权、统一认证扩展点
- `course-management`：课程、成员、资源、邀请码、课程状态
- `task-management`：任务、任务版本、评分方式、Rubric、规则
- `submission-workspace`：在线工作区、草稿、运行会话、提交受理
- `judge-orchestration`：评测任务、队列、结果、重新评测、沙箱集成
- `review-grading`：人工批改、最终成绩、发布撤回、复核
- `notification-center`：站内通知、公告、通知扩展
- `ops-audit`：课程概览、平台概览、审计日志、异常事件

## 6. 分阶段实施计划

### Phase 0：需求冻结与前置澄清

**目标**

- 将 SRS 作为 V1 基线输入，消除阻塞设计与开发的关键歧义。

**覆盖范围**

- 全局范围、V1/V1.1 分层、未决事项 `TBD-01 ~ TBD-03`

**关键动作**

- 确认 V1 只交付全部 MUST 级需求
- 明确容量目标、统一认证是否首发接入、成绩保留年限
- 确认前后端边界、评测沙箱边界、文件存储边界
- 建立需求追踪表：SRS 编号 -> 模块 -> API -> 数据实体 -> 测试用例

**输出物**

- 需求澄清记录
- V1 范围冻结清单
- 风险登记与依赖清单

**退出条件**

- 所有 MUST 级需求都有归属模块和负责人视角
- `TBD-01 ~ TBD-03` 至少形成阶段性决策或明确阻塞条件

### Phase 1：架构与工程基线

**目标**

- 在当前骨架上建立可持续开发的模块边界、工程规范和质量基线。

**关键动作**

- 完成总体架构、模块边界、领域划分和包结构约束
- 明确数据库分批设计策略、事件流和外部接口边界
- 建立统一错误模型、分页模型、审计模型、响应规范
- 完善 CI 流程：构建、测试、文档校验、基础安全扫描
- 完善开发环境：本地依赖、容器镜像、测试数据种子、质量门槛

**输出物**

- `ARCHITECTURE.md` 更新
- 总体设计 / ADR / API 设计基线
- 数据实体批次规划
- CI/CD 基线方案

**退出条件**

- 团队对包结构、实体边界、API 风格和迁移策略达成一致
- 仓库具备可重复的本地与 CI 验证路径

### Phase 2：平台配置、组织与 IAM

**SRS 覆盖**

- `FR-CFG-*`
- `FR-IAM-01 ~ FR-IAM-04`
- `NFR-SEC-01 / 03 / 04 / 06`

**关键动作**

- 落地平台配置版本模型、组织树、用户与角色模型
- 实现登录、登出、会话、账号状态、角色与资源授权
- 建立基础审计框架，覆盖登录、用户创建、角色变更等关键动作
- 设计并验证密码策略、锁定规则、会话超时

**数据批次**

- `PlatformConfig`
- `OrgUnit`
- `User`
- `AuditLog` 基础骨架

**测试重点**

- 登录成功/失败/锁定/解锁
- 未授权访问拦截
- 平台配置版本发布、回退与历史查询
- 批量导入逐行校验结果

**退出条件**

- 管理端基础治理链路可用
- 权限控制同时覆盖 UI 入口语义和服务端资源访问

### Phase 3：课程、成员与资源

**SRS 覆盖**

- `FR-CRS-01 ~ FR-CRS-05`

**关键动作**

- 建立课程、课程成员、课程资源的领域模型与状态流转
- 实现课程创建、编辑、成员管理、邀请码加入课程
- 实现课程资源上传、分类、可见范围控制
- 为课程归档和历史访问预留扩展点

**数据批次**

- `Course`
- `CourseMember`
- `Resource`

**测试重点**

- 课程状态 `draft -> active`
- 负责教师唯一性
- 邀请码有效期和刷新
- 资源权限与归属校验

**退出条件**

- 教师能创建可供后续任务挂载的课程
- 学员能成功加入课程并访问授权资源

### Phase 4：任务、评分规则与版本留痕

**SRS 覆盖**

- `FR-TSK-01 ~ FR-TSK-05`

**关键动作**

- 建立任务、任务版本、评分方式、提交规则的领域模型
- 支持编程 / 文档 / 实验报告类任务配置
- 支持开放时间、截止时间、提交形式、文件限制、评分方式配置
- 实现任务草稿、发布、关闭以及重大变更留痕

**数据批次**

- `Task`
- `Rubric` 占位骨架

**测试重点**

- 任务状态 `draft -> published -> closed`
- 开放/截止时间校验
- 混合评分权重求和校验
- 发布后重大修改留痕

**退出条件**

- 教师可以创建并发布可提交的任务
- 任务规则被后续提交链路正确消费

### Phase 5：提交受理、在线工作区与试运行

**SRS 覆盖**

- `FR-SUB-01 ~ FR-SUB-06`
- `FR-SUB-08 ~ FR-SUB-09`
- `NFR-USE-01`
- `NFR-PERF-02`
- `NFR-REL-01`

**关键动作**

- 建立在线 IDE 工作区、工作区版本、试运行会话、正式提交模型
- 实现提交入口、规则校验、唯一提交编号、提交历史查询
- 实现试运行与正式提交分离
- 为文件上传和工程快照提交建立统一受理模型
- 与文件存储建立内容引用关系，不在主库直接存大文件

**数据批次**

- `IDEWorkspace`
- `RunSession`
- `Submission`

**测试重点**

- 提交次数、时间窗口、文件大小和类型校验
- 提交受理成功后不可删除
- 试运行结果不影响成绩与正式评测
- 服务重启或前端断连后提交记录不丢失

**退出条件**

- 学员能完成“查看任务 -> 试运行 -> 正式提交”
- 教师能按权限查看课程提交记录

### Phase 6：自动评测与隔离执行

**SRS 覆盖**

- `FR-JDG-01 ~ FR-JDG-05`
- `NFR-SEC-02`
- `NFR-REL-03`
- `NFR-OBS-03`

**关键动作**

- 建立评测队列、评测作业、结果状态流转与结果持久化模型
- 接入评测沙箱或独立执行环境，确保隔离执行
- 输出结构化评测结果、日志摘要、资源指标
- 支持失败、超时、重试、降级提示
- 为重新评测和多环境扩展预留接口与状态机

**数据批次**

- `EvaluationResult`
- 评测作业 / 队列元数据

**测试重点**

- 提交后自动发起评测
- 评测状态 `pending -> running -> terminal`
- 隔离执行约束：无网络、资源上限、文件系统边界
- 评测服务不可用时的降级行为

**退出条件**

- 提交到评测到结果回写的异步链路打通
- 评测故障不阻断课程浏览、提交受理和历史结果查询

### Phase 7：人工批改、成绩生成、发布与复核骨架

**SRS 覆盖**

- `FR-REV-01 ~ FR-REV-06`
- `NFR-PRV-02`

**关键动作**

- 实现提交详情查看、教师/助教人工批改入口、评语与评分记录
- 实现最终成绩生成、发布、撤回、再次发布
- 建立成绩可追溯链路：任务 -> 提交 -> 评测/批改 -> 成绩
- 建立助教批改标识和教师复核扩展点
- 为复核申请、成绩导出、学习概览预留扩展位

**数据批次**

- `ReviewRecord`
- `Grade`

**测试重点**

- 默认以最后一次有效提交作为成绩基准
- 成绩状态 `pending -> unpublished -> published -> withdrawn`
- 成绩修改留痕
- 学员仅能查看本人反馈与成绩

**退出条件**

- 教师可完成从查看提交到发布成绩的完整链路
- 学员可查看本人反馈与成绩发布时间

### Phase 8：通知、运营概览与审计完善

**SRS 覆盖**

- `FR-NTF-01 ~ FR-NTF-02`
- `FR-OPS-01`
- `FR-OPS-03`
- `NFR-OBS-01 ~ NFR-OBS-02`

**关键动作**

- 实现站内通知、已读状态、关键事件通知触发
- 实现课程级运营概览需要的基础聚合接口
- 完善审计日志、异常事件记录、导出行为留痕
- 输出结构化日志、请求标识、用户标识和关键指标
- 为平台级概览、公告、多通道通知和异常事件增强预留扩展

**数据批次**

- `Notification`
- `AuditLog` 完整版

**测试重点**

- 关键事件通知触发与去重
- 通知已读状态管理
- 审计日志字段完整性
- 关键失败链路包含请求标识和用户上下文

**退出条件**

- 课程级运营与审计可满足 V1 核心治理需求
- 关键链路具备问题定位所需的日志与指标

### Phase 9：非功能加固与发布准备

**SRS 覆盖**

- 全部 MUST 级 NFR
- V1 验收基线

**关键动作**

- 完成性能基线测试：常规页面、提交受理、评测状态感知
- 完成可靠性验证：提交流水不丢失、评测故障隔离、备份恢复演练
- 完成安全验证：认证授权、注入/XSS/CSRF/恶意上传、凭证存储
- 完成兼容性验证：浏览器、分辨率、中文默认文案
- 完成可观测性建设：指标、日志、告警、健康检查、运行状态查看
- 完成上线清单：环境变量、密钥、回滚方案、数据初始化、审计策略

**输出物**

- 测试策略与测试报告
- 性能 / 安全 / 可靠性验证证据
- 部署与回滚清单
- 监控与告警基线

**退出条件**

- 全部 MUST 级需求具备验证证据
- 核心主链路可从平台初始化贯通到成绩查看
- 上线与回滚条件明确，可执行

### Phase 10：验收、灰度与上线

**目标**

- 完成业务验收、灰度验证、生产发布和首轮稳定期保障。

**关键动作**

- 按 SRS 第 8.2 节执行验收基线检查
- 用真实角色完成端到端走查
- 执行灰度发布、容量观察、告警联动和关键问题响应
- 将剩余 SHOULD / COULD 项整理为 V1.1 / VNext backlog

**退出条件**

- UAT 通过
- 生产可用
- 未解决问题和后续版本边界明确登记

## 7. 横切工作流

### 7.1 数据设计批次

建议按以下顺序推进逻辑实体与迁移批次：

1. `PlatformConfig / OrgUnit / User / AuditLog`
2. `Course / CourseMember / Resource`
3. `Task / Rubric`
4. `IDEWorkspace / RunSession / Submission`
5. `EvaluationResult / 评测作业元数据`
6. `ReviewRecord / Grade / Notification`

每一批都必须同时交付：

- 逻辑模型说明
- Flyway 迁移
- MyBatis-Plus 映射
- 数据访问测试
- `docs/generated/db-schema.md` 更新

### 7.2 API 与接口设计

API 设计应按业务域分批输出，而不是一次性铺满：

- 管理侧 API：平台配置、组织、用户、角色
- 教师侧 API：课程、资源、成员、任务、提交查看、批改、成绩
- 学员侧 API：课程加入、任务查看、工作区、运行、提交、结果、通知
- 集成 API：评测沙箱、文件存储、统一认证、消息通道

每批 API 都必须补齐：

- OpenAPI 文档
- 鉴权约束
- 错误模型
- 分页 / 排序 / 过滤约束
- 幂等性与审计要求

### 7.3 测试策略

测试采用分层推进：

- 单元测试：领域规则、状态机、校验逻辑、评分计算
- 接口测试：认证授权、输入输出、异常响应
- 集成测试：数据库、消息、存储、评测编排
- 端到端联调：主链路验证
- 非功能测试：性能、可靠性、安全、兼容性、恢复演练

每阶段至少包含：

- 正常路径
- 权限路径
- 边界条件
- 异常路径
- 审计与观测验证

### 7.4 文档与治理

每阶段完成时同步更新：

- `ARCHITECTURE.md`
- `docs/product-specs/*`
- `docs/generated/db-schema.md`
- `docs/security.md`
- `docs/reliability.md`
- 相关执行计划与完成记录

| M4 | Phase 7 ~ 8 | 批改、成绩、通知、审计可用 |
| M5 | Phase 9 | MUST 级 NFR 达标，具备上线条件 |
| M6 | Phase 10 | 验收通过并上线 |

## 9. 关键依赖与风险

### 9.1 外部依赖

- 评测沙箱 / 执行环境
- 文件存储服务
- 统一认证服务（如启用）
- 消息发送通道
- 前端 / IDE 客户端协同开发

### 9.2 主要风险

- 容量目标不明确会影响评测队列、资源策略和扩容方案设计
- 统一认证是否首发接入将影响账号模型和登录流程
- 在线 IDE 与评测沙箱之间的边界若定义不清，会导致运行与提交链路反复返工
- 审计、成绩追溯和权限模型如果晚做，会在后期形成高风险技术债

### 9.3 风险应对

- 在 Phase 0 关闭关键 TBD
- 在 Phase 1 完成接口边界和状态机设计
- 在 Phase 5 之前冻结运行 / 提交模型
- 在 Phase 2 起同步引入审计、安全和观测，不后置

## 10. 完成定义

一阶段可以判定为“完成”，至少满足以下条件：

- 对应 SRS 编号已覆盖
- API、数据、权限、状态流转实现完整
- 测试通过，且关键边界场景有证据
- 文档与 OpenAPI 已同步
- 监控点、审计点和错误上下文已接入
- `.\mvnw.cmd verify` 通过

1. 关闭 `TBD-01 ~ TBD-03`

3. 先完成 `Phase 2：平台配置、组织与 IAM`
