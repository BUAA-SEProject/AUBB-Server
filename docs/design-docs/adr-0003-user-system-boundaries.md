# ADR-0003：平台用户系统边界

## 状态

已采纳，2026-04-15

## 背景

AUBB 是一体化在线教学与实验平台，用户在平台治理阶段和课程教学阶段会承担不同职责。如果把“平台管理员”“教师”“助教”“学员”全部压缩成同一层全局角色，会出现三个问题：

- 平台治理权限和课程业务权限耦合，后续课程域难以演进。
- 同一用户同时承担多组织治理职责时，无法准确表达作用域边界。
- 管理员查询用户时，很难同时看清基础资料、平台治理身份和课程角色的来源。

因此需要为用户系统建立清晰的边界。

## 决策

1. 用户系统采用三层建模：
   - 用户基础资料：`users`
   - 平台治理身份：`user_scope_roles`
   - 课程成员角色：后续由课程域表独立建模
2. 当前 Phase 2 仅实现“用户基础资料 + 平台治理身份”。
3. 教师在平台治理阶段临时映射为 `CLASS_ADMIN`，仅用于平台治理，不替代未来课程域的教师角色。
4. 管理端用户查询接口返回：
   - 基础资料
   - 默认归属组织摘要
   - 账号生命周期字段
   - 平台治理身份列表
5. 用户查询与详情接口都必须按管理员作用域做服务端过滤。

## 后果

### 正向影响

- 平台治理与课程域角色边界清晰，后续课程模块可以独立建模。
- 用户查询接口更适合管理员核对归属组织、身份与账号状态。
- 账号生命周期、组织归属和治理身份来源可以被统一审计和测试。

### 代价

- 当前阶段无法直接用平台治理接口表达完整的教师/助教/学员课程关系。
- 用户响应模型会比纯基础资料更复杂，需要保持文档和 API 说明同步。

## 当前实现边界

- 课程成员角色尚未进入后端实现。
- 当前不引入外部身份源映射表，统一认证接入仍是后续扩展位。
- 用户详情默认只面向分层管理员开放，不提供普通用户的他人信息查询。
