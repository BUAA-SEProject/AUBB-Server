# ADR-0004：模块优先的模块化单体包结构

## 状态

已采纳，2026-04-15

## 背景

当前后端已经完成平台治理切片的首批落地，但此前代码主要按顶层 `api / application / domain / infrastructure` 组织。随着用户、认证、组织、配置和审计能力继续演进，这种结构会出现两个问题：

- 业务模块边界只能靠子包命名约定表达，可见性较弱。
- 新增课程、实验、任务、提交等业务时，顶层分层目录会快速膨胀，跨模块依赖也更难被察觉。

项目当前仍适合单仓单应用演进，不需要为了“模块化”立即拆分为 Maven 多模块或微服务，因此需要一套更适合模块化单体的包结构。

## 决策

1. 后端继续保持单个 Spring Boot 应用，不引入额外部署单元。
2. 业务代码统一迁入 `com.aubb.server.modules.<module>`。
3. 每个模块内部采用固定四层：
   - `api`
   - `application`
   - `domain`
   - `infrastructure`
4. 当前首批治理模块为：
   - `identityaccess`
   - `organization`
   - `platformconfig`
   - `audit`
5. 共享顶层包仅保留：
   - `com.aubb.server.common`
   - `com.aubb.server.config`
   - `com.aubb.server.infrastructure.persistence`
6. `auth`、`user`、`iam` 在当前阶段合并为 `identityaccess` 模块，不再人为拆成三个并列业务模块。

## 理由

- `identityaccess` 内部的认证、用户、平台治理身份和作用域授权天然强耦合，强拆只会增加跨模块直接依赖。
- 平台治理当前只有四个已落地业务边界，按模块目录组织可以显式表达边界，又不引入过度设计。
- 顶层只保留共享包，可以让未来课程域、实验域按同样模式继续扩展。

## 后果

### 正向影响

- 业务模块边界从目录层面变得清晰，可读性和可维护性提升。
- 后续新增课程、实验、任务、提交等模块时，有稳定的挂接方式。
- 仓库 harness 可以直接验证模块边界，而不只依赖文档约定。

### 代价

- 当前会产生一次较大的迁包与导包修复成本。
- 单应用内仍会存在少量跨模块依赖，需要后续逐步沉淀模块门面和内部约束。

## 当前实现边界

- 本次仅重构包结构和目录结构，不改变 API 路径、数据库结构和业务规则。
- 当前仍是单 Maven 模块；是否拆分为多模块构建留待后续按团队规模和发布节奏再评估。
- 课程、任务、提交、评测等后续模块仍处于扩展位，尚未落地到仓库代码。
