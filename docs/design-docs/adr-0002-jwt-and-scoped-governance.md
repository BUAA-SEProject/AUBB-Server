# ADR-0002：JWT 认证与分层治理身份

## 状态

已采纳，2026-04-14

## 背景

Phase 2 首版实现采用了浏览器 Session、全局平台角色和平台配置版本化模型。最新需求对这三部分提出了新的约束：

- 认证必须改为 JWT
- 平台治理必须以 `学校 -> 学院 -> 课程 -> 班级` 为固定组织层级
- 每一层都存在对应管理员，且一个用户可以同时拥有多个身份
- 平台配置修改后必须立即生效，不再保留版本化流程

原有设计已无法支撑新的身份模型和授权边界，因此需要显式收敛新的治理基线。

## 决策

1. 使用 Spring Security Resource Server 校验 JWT，应用本地签发 HS256 访问令牌。
2. 将管理权限从全局角色改为 `user_scope_roles` 作用域身份分配模型。
3. 组织树固定为四层：
   - `SCHOOL`
   - `COLLEGE`
   - `COURSE`
   - `CLASS`
4. 管理身份固定为：
   - `SCHOOL_ADMIN`
   - `COLLEGE_ADMIN`
   - `COURSE_ADMIN`
   - `CLASS_ADMIN`
5. 教师在平台治理阶段映射为 `CLASS_ADMIN`。
6. 平台配置改为单份实时配置 `platform_configs`，通过更新覆盖当前生效配置。

## 当前实现边界

- 平台配置管理和审计查询只开放给学校管理员。
- 首个学校节点和首个学校管理员通过默认关闭的启动期 bootstrap 建立。
- 不在本阶段实现刷新令牌、令牌撤销名单或统一认证接入。

## 后果

### 正向影响

- 认证改为无状态，更适合 API 化和后续前后端分离。
- 身份模型可以自然表达“一人多岗、多组织”的治理需求。
- 平台配置流程更简单，减少了治理误用和实现复杂度。

### 代价

- 旧 Session 接口、测试和文档需要整体重写。
- 学院/课程/班级管理员的细粒度授权必须在应用层增加作用域校验。
- 若未来需要平台配置历史，需额外引入审计快照或配置变更历史表。

## 备选方案

### 方案一：继续使用 Session，只增加多身份

未采用。原因是用户已明确要求改为 JWT，而且双模型并存只会放大维护成本。

### 方案二：保留配置版本化，仅在编辑后自动发布

未采用。原因是这仍然保留了版本发布心智，与“修改立刻生效”的要求不一致。

### 方案三：继续使用全局平台角色，再通过组织归属做补充判断

未采用。原因是无法正确表达一个用户在多个组织层级同时具备不同管理职责的场景。
