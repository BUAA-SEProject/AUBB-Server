# ADR-0005：课程系统第一切片采用聚合式 `course` 模块

## 状态

已采纳，2026-04-15

## 背景

`design/courses.md` 将课程系统拆分为多个细粒度子模块，例如课程模板、开课实例、教学班、成员、规则、内容和查询模块。这一方向适合长期演进，但当前仓库仍处于单应用、单仓、主链路优先阶段。如果一次性把课程系统拆成过多并列模块，会带来三个问题：

- 首个课程切片的交付面过大，难以在当前阶段稳定落地。
- 教学角色、组织联动、成员导入和班级配置会被拆散到过多目录，影响实现效率。
- “学生禁止自主选课、教师从系统用户名单批量添加/导入”这一新业务约束需要先收敛到一个清晰实现。

## 决策

1. 当前课程系统先落在单个 `com.aubb.server.modules.course` 聚合模块中。
2. 本轮课程切片优先实现：
   - 学期
   - 课程模板
   - 开课实例
   - 跨学院共同管理映射
   - 教学班
   - 课程成员
   - 班级功能开关
3. 教师、助教、学生的课程角色统一由 `course_members` 承担，不复用平台治理身份。
4. 学生自主选课本轮不实现；课程成员仅允许教师批量添加或导入既有系统用户。
5. 助教权限先采用固定边界：
   - 可查看被授权教学班成员
   - 不可修改课程核心信息
   - 不可开关班级功能
   - 不可批量添加成员

## 理由

- 聚合式 `course` 模块更适合当前单体阶段的首个课程闭环交付。
- 课程业务权限与平台治理身份继续分离，符合已有 ADR-0003。
- 学生禁止自助选课的要求会直接影响接口和状态机，必须在首轮实现中显式固化。

## 后果

### 正向影响

- 课程第一切片可以在当前仓库内快速形成可验证主链路。
- 后续任务、资源、实验、成绩模块可以围绕 `course_offerings`、`teaching_classes`、`course_members` 挂接。
- 教学角色边界变得明确，不再依赖平台治理身份临时代偿。

### 代价

- `course` 模块内部会先承载较多子能力，后续可能需要再向更细的课程子模块拆分。
- 课程资源正文、规则引擎、ABAC 和事件 outbox 继续保留为扩展位，本轮未实现。
