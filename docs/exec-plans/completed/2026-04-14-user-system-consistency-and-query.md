# 执行计划：用户系统一致性校对与查询能力补齐

## 目标

在当前 Phase 2 治理切片基础上，完成用户系统的文档一致性校对，并补齐最小可用的“用户查询”管理能力，使管理员能够查看自己作用域内的用户并核对账号状态、身份和归属组织。

## 范围

- 仓库内用户系统相关代码、测试与文档
- `../docs` 中与用户系统、平台治理、认证和用户手册直接相关的文档

## 不在范围

- 密码重置、首次改密、统一认证接入
- 课程成员管理与课程级角色模型
- 平台概览、通知和后续课程域能力

## 关键差异

1. 当前实现缺少用户查询接口，难以支撑“组织用户”与“检查账号状态”的管理场景。
2. 上级文档中仍有部分旧口径：
   - 用户导入流程仍写为 Excel / `.xlsx`
   - 用户手册仍保留“保存草稿并发布生效”和“检查浏览器 Cookie”等旧表述
3. 仓库实现已经收敛为 JWT、单份即时生效平台配置和 CSV 导入，需与文档统一。

## 计划动作

1. 校对仓库实现与 `docs/`、`../docs/` 的用户系统口径差异。
2. 先补失败测试，覆盖用户列表查询、作用域过滤和导入类型约束。
3. 实现用户查询接口与必要的授权/过滤逻辑。
4. 更新仓库与上级文档，明确当前实现和后续扩展位。
5. 运行针对性测试与全量 `./mvnw test` 验证闭环。

## 风险

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 用户查询若不做作用域过滤，可能泄露跨组织数据 | 安全边界失效 | 复用现有治理授权模型，默认按主组织作用域过滤 |
| SRS 与当前实现对导入格式口径不同 | 文档持续冲突 | 在上级文档中写明当前实现为 CSV，保留未来扩展到 `.xlsx` 的位置 |
| 扩展查询接口后，返回结构若与现有用户创建接口耦合过深 | 后续演进困难 | 尽量复用现有 `UserView`，保持接口最小增量 |

## 验证路径

- 接口测试：
  - 学校管理员可分页查询用户
  - 学院管理员只能看到本作用域内用户
  - 超出作用域的组织筛选请求被拒绝
  - 非 `csv` 的导入类型被拒绝
- 一致性验证：
  - 仓库文档与 `../docs` 的用户系统口径一致
  - `RepositoryHarnessTests` 文档链接检查通过
- 全量验证：
  - `./mvnw test`

## 退出条件

- 管理员可在自身作用域内查看用户列表与用户详情
- 用户系统、认证和导入方式的仓库内外文档口径一致
- 相关接口测试和仓库 harness 校验通过
