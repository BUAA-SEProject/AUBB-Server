# 执行计划：judge 第一切片

## 目标

在现有 assignment、submission 和对象存储基础上，落地 `judge` 模块的第一切片：为正式提交自动创建评测作业，提供评测作业状态持久化、学生/教师查询，以及教师手动重新排队入口，为后续 go-judge 集成和结果回写提供稳定骨架。

## 范围

- `modules.judge` 的数据模型、应用服务、控制器与基础枚举
- 正式提交后自动创建 `judge_jobs`
- 学生按提交查看自己的评测作业
- 教师按提交查看评测作业，并手动重新排队
- Flyway 迁移、产品规格、数据库结构与架构同步
- judge 相关集成测试

## 不在范围

- 真正调用 go-judge 执行代码
- 评测状态 `running -> terminal` 的执行器回写
- 运行日志、资源占用、测试用例详情
- 成绩计算、人工批改、成绩发布

## 风险

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 过早把执行器细节写死到业务层 | 后续 go-judge 接入时返工 | 先只固化评测作业模型和状态机，执行器作为后续基础设施扩展位 |
| 提交服务直接依赖评测实现细节 | submission 与 judge 强耦合 | 由 judge 模块提供“入队”应用服务，submission 只调用稳定入口 |
| 没有明确的重评测入口 | 后续教师侧操作再返工 | 第一切片先补教师重排队接口，保持作业粒度清晰 |

## 验证路径

- `./mvnw spotless:apply`
- `./mvnw -Dtest=SubmissionIntegrationTests test`
- `./mvnw clean verify`

## 决策记录

| 决策 | 原因 |
| --- | --- |
| 第一切片先只建评测作业骨架，不直接打 go-judge | 先打通“提交 -> 评测作业持久化”主链路，再接执行器更稳 |
| 评测作业默认在提交后自动创建 | 对齐主链路目标，避免后续补自动入队时破坏现有提交契约 |
| 教师重排队只创建新的 pending 作业，不修改历史作业 | 保持评测可追溯性和历史留痕 |
