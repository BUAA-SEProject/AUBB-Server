# Phase 2：平台配置、组织与 IAM

> 注意：本记录反映的是较早一轮 Phase 2 方案，包含后续已被放弃的 Session 认证与平台配置版本化口径。
> 当前有效的最终收敛记录见 [2026-04-14-jwt-scope-governance-refactor.md](2026-04-14-jwt-scope-governance-refactor.md)。

## 目标

在 AUBB-Server 中完成开发总计划的 Phase 2，交付平台治理与身份认证的首个真实业务切片，使平台管理员可以完成平台初始化、组织维护、用户导入、角色与账号状态管理，并使用户可以完成登录、退出和受保护资源访问。

## 关键设计决策

| 决策 | 说明 |
| --- | --- |
| 认证方式 | 采用浏览器 Session 认证；控制器通过当前会话识别用户 |
| 角色模型 | 平台角色采用显式表结构与服务端 Authority 映射；课程角色后续单独建模 |
| 持久化 | 以 PostgreSQL + Flyway + MyBatis-Plus 落地，所有结构变更通过迁移脚本管理 |
| 审计 | 以应用服务显式写入为主，优先覆盖登录、配置发布/回退、用户导入、角色和状态变更 |
| 导入能力 | 首版提供可扩展的批量导入能力，并返回逐行校验结果 |
| 运行时基线 | 主应用重新接入 DataSource 与 Flyway；本地通过 Docker Compose，测试通过 Testcontainers 保证可重复 |

## 风险

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 引入数据库后改变当前“无依赖启动”基线 | 影响本地启动和 verify 路径 | 文档明确新的启动方式，测试使用 Testcontainers，保留公开健康检查 |
| 平台角色与后续课程角色边界不清 | 后续课程域可能返工 | 本阶段只实现平台角色，课程角色在文档中定义扩展边界 |
| 批量导入格式选择与系统级设计不完全一致 | 影响前后端对接预期 | 在产品规格与 API 文档中明确首版导入格式与扩展点 |
