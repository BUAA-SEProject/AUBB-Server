# 2026-04-16 作业模块能力重规划与阶段推进

## 目标

基于当前仓库真实实现，重新校准作业模块路线，使其与“按课程或班级发布作业、结构化多题型、题库与组卷、多次提交、自动评测与人工批改、成绩发布与反馈、编程题在线 IDE 与多语言评测”这一目标保持一致，并为后续连续开发提供稳定的阶段边界。

## 范围

- 盘点 `assignment / submission / judge / grading` 四个模块的当前覆盖度
- 明确已完成能力、部分完成能力与缺口
- 固化最佳职责拆分与兼容约束
- 给出下一阶段的推荐推进顺序

本计划当前不直接落具体业务代码，而是作为后续多个切片的总执行计划。

## 当前状态盘点

### 已完成

- 作业已支持按课程实例或教学班发布，并有 `openAt / dueAt / maxSubmissions`
- 结构化试卷已支持多个大题与以下题型：
  - `SINGLE_CHOICE`
  - `MULTIPLE_CHOICE`
  - `SHORT_ANSWER`
  - `FILE_UPLOAD`
  - `PROGRAMMING`
- 已支持题库创建、列表、详情和引用组卷
- 已支持正式提交、版本记录、附件关联、客观题自动评分摘要
- 已支持教师 / 助教人工批改、assignment 级成绩发布、教师侧成绩册第一阶段
- 已支持 go-judge 样例试运行、question-level judge、`STANDARD_IO` 和第一阶段 `CUSTOM_SCRIPT`

### 部分完成

- 题库生命周期还缺更新、删除、标签、检索和更强的组卷编辑体验
- 编程题后端当前已支持 `entryFilePath + files + artifactIds` 的目录树快照，但前端目录树交互与逐文件操作仍未完成
- 多语言运行时虽然已有 `PYTHON3 / JAVA17 / CPP17` 模型与部分执行链路，但验证和日志一致性不足
- 成绩发布已打通教师侧和学生侧第一阶段，但成绩导出与更完整统计仍缺失
- 评测日志已有摘要、`stdout/stderr` 和用例结果，但还缺更可复现的完整执行元数据

### 当前优先级

1. 在线 IDE 第二阶段
2. 多语言运行时稳定化
3. 题库与组卷第二阶段
4. 成绩与反馈第二阶段
5. 判题日志与可复现性第二阶段

## 风险

- 工作区从“单正文 + 附件”升级到目录树后，既有 API 兼容和数据迁移复杂度会明显上升
- 多语言稳定化会放大 go-judge 运行镜像、编译过程和错误日志格式的一致性要求
- 若过早开放自定义脚本，容易引入运行安全、调试复杂度和文档失真问题
- 成绩导出和多作业加权若先于已固定的学生成绩可见性边界继续外扩，会导致权限边界重新变得混乱

## 验证路径

- 每个后续切片都必须先补失败的集成测试，再做最小实现
- 每个切片至少执行一次 `./mvnw spotless:apply`
- 每个切片至少执行对应专项测试
- 阶段收口执行 `./mvnw clean verify`
- 相关产品规格、数据库结构、架构文档与 `todo.md` 必须同步

## 决策记录

- 保持“模块优先、模块内分层”结构，不新增与现有职责重叠的平级业务模块
- 继续用 `assignment` 管试卷快照，而不是让 `submission` 或 `judge` 反向持有题目定义
- 在线 IDE 先补工作区目录树与多文件模型，不直接跳到开放式远程开发环境
- 多语言 V1 先固定三种语言：`PYTHON3`、`JAVA17`、`CPP17`
- 学生成绩面板与导出放在 `grading` 第二阶段，不把统计逻辑散落到 `submission`
